# HootoDay Project Notes

## プロジェクト概要

- プロジェクト名：HootoDay
- 完全な個人利用専用
- スケジューラーと健康記録を統合したアプリ
- 最初はブラウザ版として開発
- 将来はPWA化、iPhone対応、デスクトップアプリ化を検討
- 配布、一般公開、複数ユーザー対応、ストア公開の予定なし

## 主要機能

### 1. スケジューラー

- 月間カレンダー
- 日付ごとの予定
- 開始時刻、終了時刻、終日予定
- 予定カテゴリ
- 短い日記・メモ
- 日記やメモがある日のカレンダー表示

### 2. 健康記録

- 睡眠
- 体重
- 朝食、昼食、夕食、間食
- 運動
- 体調メモ
- 各項目の専用閲覧ページ
- 週、月、半年、年単位の集計
- TXTまたはMarkdown出力
- JSONバックアップと復元

## 健康記録の概要

### 睡眠

- 就寝時刻と起床時刻から睡眠時間を計算する
- 途中覚醒は複数登録可能とする
- 覚醒時刻だけの場合は、初期値として15分を睡眠時間から差し引く
- 覚醒開始時刻と終了時刻がある場合は、実際の覚醒時間を睡眠時間から差し引く

### 体重

- 最新値、目標値、標準体重、美容体重、期間平均、グラフを将来表示する

### 食事

- 朝食、昼食、夕食、間食の各欄に自由記述で記録する

### 運動

- 1日に複数回登録可能とする
- 運動種類、時間、心拍数、推定消費カロリー、メモを記録する

### 体調

- 全体、膝、腰、生理・周期、気になることを記録する

## 開発方針

- 一度に大規模実装をしない
- Phase単位で実装、確認、記録、commitする
- データを壊す可能性がある変更では、事前にバックアップを用意する
- 自分専用のため複数ユーザー機能は作らない
- 入力の手軽さと見やすさを優先する
- PCとiPhoneでは将来別UIにしてよい
- 通知とiPhoneウィジェットは後期Phaseへ回す
- 健康データの計算値は医療診断ではなく、本人用の概算記録とする

## 将来追加機能：即売会・在庫記録

### 機能の目的

- 即売会イベントごとに、商品別の持込数と販売結果を記録する
- 単なる自由記述メモではなく、数量と金額を数値として保存・集計する
- 年2回程度の利用でも、過去イベントとの比較や累計を確認できるようにする
- 完全な個人利用専用機能とする

### イベント単位の記録項目

- イベントID
- イベント名
- 開催日
- 会場
- 全体メモ
- 商品別記録
- 作成日時
- 更新日時

### 商品別の基本項目

- 商品ID
- 商品名
- 持込数
- 売れた数
- サンプル配布数
- 残数
- 単価
- 売上
- 商品別メモ

### 自動計算と入力条件

- 残数 = 持込数 - 売れた数 - サンプル配布数
- 売上 = 売れた数 × 単価
- サンプル配布数は売上に含めず、在庫だけを減らす
- 計算結果が負数になる入力は保存しない

### 集計したい内容

- イベントごとの総持込数
- イベントごとの総販売数
- イベントごとの総サンプル配布数
- イベントごとの総残数
- イベントごとの総売上
- 商品ごとの累計販売数
- 商品ごとの累計サンプル配布数
- 商品ごとのイベント後残数
- 年ごとの販売数と売上
- 前回イベントとの比較
- 過去イベント一覧

### HootoDayとの連携案

- カレンダーへ即売会予定を登録する
- 即売会予定または選択日詳細から「即売会記録を開く」
- 日記・メモ欄には当日の感想を書き、数量と売上は即売会記録へ分離する
- 将来「記録を見る」からイベント別・商品別集計を開く
- 将来TXT、Markdown、CSVなどへ出力できるようにする

### 将来拡張の余地

初期実装の必須項目にはしないが、データ構造上は次の項目を追加できる余地を残す。

- 追加生産数
- 関係者への配布
- 破損・廃棄数
- 通販分
- 自宅に残した在庫
- 商品画像
- 原価と利益

### Special Phase：即売会・在庫記録

- 健康記録や出力機能とは混ぜず、独立したPhaseとして扱う
- 実装時期は後で決定する
- 初期実装はイベント、商品、持込数、販売数、サンプル配布数、残数、単価、売上を対象とする
- 高度な在庫移動、追加生産、原価・利益計算は後期拡張とする
- 既存の日記・予定・健康記録とは別データとして保存する

- 復元直前に全対象localStorageキーの元値とキー未存在状態をメモリへ退避

### HootoDay Sync SQL Design Phase（設計時点：PRECHECK・適用・rollback・検証SQL作成済み、Supabase未適用）

- Supabase適用前の最新JSONバックアップ取得済み
- DayMemo限定の`hooto_day_sync_records`を設計
- operation IDは別の`hooto_day_sync_operations`で管理し、適用結果・競合結果の同一再送を冪等化
- request全体の正規化JSONBを組み込みMD5でfingerprint化し、同じoperation IDの異なる内容を拒否
- operation履歴は無期限保存せず推奨30日。今回は自動cleanupを追加せず、cleanup後は該当IDの冪等保証が失われる
- 専用sequenceの`change_sequence`をpull cursorの正本とし、成功mutationのcommit順をadvisory lockで直列化
- contentは一般的なASCII・Unicode空白を含めてtrim済み・非空をDBでも検証
- revisionとbase revision一致による競合検出、payload NULLのtombstoneを採用

- RPCのsearch_pathを固定し、authenticatedだけへEXECUTEを許可。PUBLIC・anonはREVOKE
- 空配列やレコード不在を削除と解釈せず、空iPhoneは初回pullのみとする保護方針を明記
- rollback SQLは今回のHootoDay専用オブジェクトだけを削除
- PRECHECK SQLで適用前の共通RPC hash・共通構造署名をCSV保存し、適用後VERIFYと比較
- verify SQLはsequence・fingerprint・権限の構造確認とauthenticated clientでの実操作テストを分離
- 既存workspace/member/pairing基盤と既存RPCは変更なし
- `app_workspace_state`と`current_hooto_sync_key_hash()`は不使用
- localStorage version 1、JSON backup formatVersion 2・旧v1互換を維持
- Service Workerは未導入、在庫・販売同期は対象外
- 次はSQLレビューとSupabaseへの手動適用
- commitはユーザー確認後
- PRECHECKを`BEGIN READ ONLY`で囲み、共通テーブルの詳細カラム属性を曖昧性のないJSONB署名へ補強
- PRECHECK比較は取得対象の署名・RPC hashの一致確認であり、全DB要素や実効権限の完全同一証明ではないことを明記
- fingerprint内ではSQL NULLがJSON nullへ変換される点と、有効入力の事前検証・operation kind分離で正当なrequest間の曖昧性を防ぐ設計を明記
- APPLY postflightをsequence属性・ACL、RLS・policy、table権限、change sequence、operation管理、RPC属性・権限までfail-closedで検査するよう補強
- VERIFYとanon/authenticated client実操作でrole継承を含む実効権限を最終確認する方針を維持
- この設計時点では、SupabaseへのSQL実行・構造変更は未実施
